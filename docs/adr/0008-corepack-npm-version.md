# ADR-0008: Pin npm version via packageManager field and Corepack

**Status:** Accepted  
**Date:** 2026-06-21

## Context

After ADR-0007 pinned Node.js to v24, Cloudflare Pages still used npm 10.9.2 from its build image rather than the npm 11.x that ships with Node 24. This caused `npm ci` to fail with:

```
Missing: @emnapi/runtime@1.11.1 from lock file
Missing: @emnapi/core@1.11.1 from lock file
```

## Root cause

`@tailwindcss/oxide-wasm32-wasi` (a WASM fallback for Tailwind's native Rust module) declares `cpu: ['wasm32']`. On an x64 Linux build machine:

- **npm 11**: skips installing the wasm32 package (native x64 binary used), and does not record its dependencies (`@emnapi/runtime`, `@emnapi/core`) in the lock file
- **npm 10**: sees the wasm32 package listed in the lock file (as `optional: true`), expects its regular dependencies to also be recorded, finds them absent, aborts `npm ci`

The lock file generated by npm 11 is correct for npm 11 but incompatible with npm 10's stricter validation.

## Decision

1. Add `"packageManager": "npm@11.6.2"` to `package.json`
2. Set `ENABLE_EXPERIMENTAL_COREPACK=1` as an environment variable in the Cloudflare Pages dashboard (Production and Preview environments)

## Rationale

Cloudflare Pages activates Corepack when `ENABLE_EXPERIMENTAL_COREPACK=1` is set. Corepack reads the `packageManager` field and downloads/activates the specified npm version for the install step, replacing the build image's default npm 10.

## Consequences

- `npm ci` on Cloudflare Pages now uses npm 11.6.2, matching the lock file format
- `ENABLE_EXPERIMENTAL_COREPACK=1` is a dashboard setting — it is NOT committed to the repo and must be manually set per Cloudflare Pages project
- When upgrading npm locally, update the `packageManager` field to match and create a new ADR
- The `packageManager` field also causes Corepack to prompt or warn local developers who have a different npm version active
